Bit System

O que começou em 2025 como um design system interno para acelerar entregas da agência evoluiu para um sistema operacional de marca com runtime em código, catálogo de assets, recipes reutilizáveis, documentação humana e uma camada própria para IA.

Ver sistema online Ver origem no Figma

O problema original

O sistema nasceu para parar de refazer decisões visuais e operacionais em toda entrega.

Na Dois Palitos Growth Lab, cada novo site, landing page ou material comercial acabava reabrindo discussões que já deveriam estar resolvidas. Cor, hierarquia, espaçamento, estrutura e assinatura de marca voltavam a ser definidos projeto após projeto.

A primeira versão do Bit System surgiu em 2025 para resolver isso como problema de produto e operação. A meta inicial era simples: criar uma base confiável para construir páginas mais rápido, com menos retrabalho e mais consistência.

Depois, com novas rodadas de iteração e com ferramentas como Claude Code e Codex acelerando exploração, documentação e implementação, o projeto deixou de ser apenas uma biblioteca visual interna. Ele virou um produto mais completo, com contrato estrutural, runtime em código, governança de assets e uma interface explícita para IA.

Como evoluir um design system criado para ganho operacional em 2025 para um sistema que também sirva como contrato de marca, runtime estável, catálogo de assets e contexto confiável para humanos, devs e IA?

O que precisava mudar

  • 1

    O sistema inicial resolvia produção, mas não separava contrato, runtime e referência. Isso limitava manutenção, clareza e consumo técnico.

  • 2

    Quem consumia o sistema dependia demais do Figma ou das páginas de showcase. Isso funcionava para pessoas, mas não era a melhor interface para código nem para IA.

  • 3

    Os assets precisavam de governança mais explícita. Logo, símbolo, ícones e legados precisavam ter catálogo canônico e regra operacional.

  • 4

    IA generativa precisava de contexto claro para não adivinhar o sistema. Sem trust order, contratos e guias, a deriva seria inevitável.

Dois ciclos, o mesmo sistema

2025

Primeira versão, foco operacional

A primeira camada do Bit System foi construída para resolver velocidade e consistência dentro da operação. O foco estava em fundamentos, componentes, blocos prontos, brand e acessibilidade, com uso real no fluxo de criação de páginas da agência.

2026

Novo ciclo, foco em arquitetura e interoperabilidade

O sistema foi endurecido em código. Entraram `tokens.json`, `tokens.css`, `components.css`, catálogo canônico de assets, sprite real de ícones, documentação mais clara e uma pasta `ai/` desenhada para modelos diferentes consumirem o sistema sem depender de HTML de vitrine.

Iteração

IA como aceleração de manutenção, não como efeito de marketing

Ferramentas como Claude Code e Codex ajudaram a acelerar leitura estrutural, documentação, validação e refino do runtime. O ganho não foi só produzir mais rápido. Foi conseguir iterar o sistema com mais profundidade e menos atrito entre decisão, implementação e governança.

A iteração mais recente não ficou só em documentação e naming. O sistema hoje existe como produto online navegável, com páginas específicas para foundations, componentes, recipes e consumo por IA. Isso deixa o valor do Bit System mais visível, ensinável e mais fácil de consumir por públicos diferentes.

O que o Bit System é hoje

Marca

Sistema operacional de marca

O Bit System organiza linguagem visual, assinatura, hierarquia, tom e regras de uso para site, landing page, proposta, relatório, deck e social media.

Runtime

Base pública em CSS

`tokens.css` e `components.css` funcionam como entrypoints públicos. Isso reduz ambiguidade e permite consumo técnico com menos dependência de contexto implícito.

Assets

Catálogo canônico de logos e ícones

`assets/manifest.json` e `assets/icons.svg` passam a definir escolha operacional, ids reais e fronteira entre atual, legado e contexto recomendado.

IA

Camada própria para consumo por modelos

A pasta `ai/` entrega leitura curta, contratos JSON, trust order e regras explícitas para que modelos diferentes montem peças sem inventar o sistema.

Arquitetura de uso

A principal mudança de maturidade foi separar melhor o que é contrato, o que é runtime e o que é referência visual. Isso deixa o sistema mais claro para manutenção, implementação e consumo por ferramentas diferentes.

tokens.json

Contrato estrutural dos tokens

Define nomes, relações semânticas, temas e base estrutural do sistema. É a referência de nível mais alto para o domínio de tokens.

tokens.css

Entrypoint público do runtime de tokens

Mantém o consumo simples para qualquer página e centraliza a camada de runtime que espelha o contrato estrutural.

components.css

Entrypoint público de layout, UI e recipes

Concentra grid, componentes e composições reutilizáveis. É a superfície pública para consumo por dev e por exemplos da agência.

assets/manifest.json

Catálogo canônico de assets

Escolhe logo, símbolo, ícones e contexto recomendado. Evita deriva entre peça nova, legado e runtime real.

assets/icons.svg

Implementação real dos ids de ícone

Os ids usados pelo sistema deixam de ser implícitos. Isso melhora consistência técnica e elimina parte do ruído histórico.

docs/ + README.md

Camada humana de onboarding e governança

Explica arquitetura, uso, limites, manutenção e checklist operacional sem exigir leitura profunda do CSS para entender o sistema.

Camada própria para IA

A pasta `ai/` foi criada para que modelos diferentes consigam entender o sistema sem depender de conhecimento específico do repositório. Em vez de pedir que a IA adivinhe o Bit System a partir de exemplos soltos, o projeto entrega uma interface curta, legível e interoperável.

ai/llms.txt

Entrada curta

Define o sistema, a ordem preferida de leitura e as regras mais críticas para consumo rápido por modelos genéricos.

ai/system.json

Índice do pacote e trust model

Explicita versão, arquivos relevantes, source of truth e trust order, reduzindo ambiguidade sobre o que deve ser lido primeiro.

ai/components.json / recipes.json

API canônica do sistema para IA

Expõe blocos, recipes e composições de marca em formato mais confiável para automação, RAG e modelos que trabalham melhor com JSON do que com CSS.

ai/templates.json / assets.json

Contextos de saída e escolha de ativos

Diz qual template usar, que recipe priorizar, qual logo escolher e em que contexto o sistema está mais maduro.

  1. 1
    Ler `ai/llms.txt` e `ai/system.json` Entender o sistema, a ordem de leitura e no que confiar primeiro.
  2. 2
    Aplicar `ai/decision-rules.json` Seguir regras como dark first, tipografia, limite de CTA neon e preferência por recipes.
  3. 3
    Escolher contexto, blocos e assets Usar `templates.json`, `components.json`, `recipes.json` e `assets.json` antes de descer para precisão mais baixa.
  4. 4
    Descer para `tokens.json`, manifest e CSS somente quando necessário Esse fluxo reduz erro, subjetividade e deriva entre o sistema documentado e a peça gerada.

Validação e maturidade atual

Hoje eu descreveria o Bit System como forte enquanto sistema de marca, bom enquanto sistema operacional interno e bom para consumo por IA séria. Ele já tem fonte de verdade por domínio, runtime modularizado, catálogo canônico de assets e scripts de validação para reduzir quebra estrutural.

Tempo de produção 30 min por página nova, contra cerca de 4 horas antes da primeira versão do sistema
Adoção 100% de adoção interna nos projetos que usam a base operacional da agência
Validação 2 validadores passados no main para integridade do sistema e do pacote de IA
Acessibilidade WCAG AA como base de contraste e acessibilidade aplicada aos tokens atuais
scripts/validate-bit-system.ps1 scripts/validate-ai-package.ps1

Estado atual

Já é um sistema maduro o bastante para uso real

A base atual já consegue sustentar produção humana, implementação em código, escolha de assets e leitura por IA com menos dependência de contexto informal.

Limite honesto

Ainda não é 100 por cento auto derivado

O principal limite restante é que o CSS ainda não é gerado automaticamente a partir de `tokens.json`, e alguns exemplos ainda carregam exceções manuais.

O que este projeto prova

1

Consigo pensar design system como infraestrutura de operação. O valor do Bit System não está só em componente bonito. Está em reduzir retrabalho, alinhar produção e sustentar consistência em contextos reais de agência.

2

Consigo iterar um sistema além da primeira versão. O projeto não parou no Figma de 2025. Ele foi reestruturado até ganhar runtime, governança, contrato estrutural e superfície de consumo mais madura.

3

Sei aproximar marca, código e IA no mesmo sistema. Aqui, branding não ficou separado de implementação. E IA não entrou como camada cosmética, mas como consumidora real do sistema com trust order e contratos explícitos.

4

Meu repertório vai além da interface final. Este case mostra arquitetura, documentação, naming, governança, manutenção e capacidade de transformar um sistema interno em produto operacional vivo.
